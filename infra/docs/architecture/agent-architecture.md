# Agent Architecture

This is the **agent reasoning plane** ‚Äî what happens *inside* one `InvokeAgentRuntime` call once the CUSTOM_JWT authorizer has let it through: how `build_agent()` assembles the system prompt from preloaded doctrine plus an on-demand skills catalog, how the Strands `Agent` loops between **Bedrock Nova Lite** (`ConverseStream`) and its tools, how the three knowledge surfaces are wired (ontology bindings via `describe_entity`, on-demand skill bodies via `load_skill`, and the **Knowledge Base** via `search_policies`), and which tools are **LOCAL** (never traverse the Gateway) versus served by the **AgentCore Gateway (MCP)** under Cedar authorization. The grey **Observability** band is the control plane (telemetry, not request flow). The identity/OBO/Snowflake-RLS **data plane** and the build/publish/deploy pipeline are intentionally omitted ‚Äî see `infra/docs/architecture/data-plane.md` for that request/data plane.

**Legend** ‚Äî official AWS icons, left ‚Üí right. Edges: **solid dark** = request / data path ¬∑ **blue dashed** = identity / token / secret ¬∑ **grey** = supporting (incl. telemetry); primary steps are numbered. Rounded boxes are trust / responsibility zones. The diagram is generated from [`specs.json`](specs.json) by the `architecture-skill` skill ‚Äî edit the spec, not the SVG.

## How to read it

The walkthrough below is the full turn in depth ‚Äî **steps 1‚Äì6** assemble the agent (`build_agent()`); **steps 7‚Äì13** are the per-turn reasoning loop the diagram above shows.

**1‚Äì2 ¬∑ Entrypoint ‚Äî the CUSTOM_JWT gate, then code runs.** The caller issues `InvokeAgentRuntime` with `{prompt, session_id}` and a Bearer Entra user JWT. The Runtime Endpoint's CUSTOM_JWT authorizer validates `aud`/`iss` *before any code runs* ‚Äî `allowed_audience = ["api://${entra_agent_app_id}"]` on `aws_bedrockagentcore_agent_runtime_endpoint.default` in `infra/terraform/runtime.tf`. Only then does `BedrockAgentCoreApp`'s `@app.entrypoint async invoke(payload, context)` execute (`agent/src/order_triage/runtime.py`). It reads `prompt` from `payload["prompt"]`/`inputText` and `session_id` from the payload, falling back to `context.session_id`. The runtime streams back **NDJSON** ‚Äî each `yield` is one line: answer/thinking text bare, everything else routed through `step_events`.

**3 ¬∑ Identity ‚Üí the memory partition key (NOT authorization).** `_extract_user_jwt(context)` reads the allow-listed `Authorization` header (`request_header_allowlist = ["Authorization"]`, `runtime.tf`), strips the `Bearer ` prefix, and `identity.set_user_jwt` stores it in a ContextVar for the turn (`agent/src/order_triage/identity.py`). `_subject_from_jwt` base64-decodes the JWT payload **without verifying the signature** (the authorizer already verified it) to read `sub`, falling back to `oid`; `actor_id()` returns that subject or the shared `ANONYMOUS_ACTOR = "order-triage"`. This subject is used **only** as the Memory `actor_id` partition ‚Äî never to authorize a call.

**4 ¬∑ Gateway MCP client opens for the whole turn.** `build_gateway_client` (`agent/src/order_triage/gateway.py`) builds a Strands `MCPClient` over `streamablehttp_client(GATEWAY_URL)` with `headers={"Authorization": f"Bearer {ident.raw_jwt}"}` ‚Äî it forwards the **same inbound user JWT**; the agent mints no credentials. It returns `None` (which `runtime.py` treats as a hard `RuntimeError` ‚Äî this runtime is Gateway-only) when `GATEWAY_URL` or the user identity is missing. `runtime.py` enters `with gw_client:` and calls `gw_client.list_tools_sync()` to fetch the backend tool surface as `extra_tools`. The MCP session must stay open for the whole stream because the tools are only callable while the session is live.

**5 ¬∑ System-prompt assembly ‚Äî the three knowledge surfaces wired in.** `build_agent` (`agent/src/order_triage/agent.py`) assembles `SYSTEM_PROMPT` from three parts: **(a)** hard-coded order-triage guardrail text (flag only on evidence ‚Äî a high risk score, a policy requirement, or a confirmed SAP hold ‚Äî and only on an OPEN order; only `orders___flagOrder` truly flags; ground every policy claim in `search_policies`); **(b)** `_DOCTRINE` = the joined `.body` of all `preload:true` skills via `skill_loader.preloaded_skills()` ‚Äî today just `using_the_knowledge_layer.skill.md`, which teaches the Ontology‚ÜíSkills‚ÜíKB hierarchy; **(c)** `skill_loader.skills_catalog()`, the on-demand catalog (name, description, applies-to entities/actions) of the **NON-preload** skills the model can `load_skill(name)` to read in full. `skills_catalog` explicitly excludes preloaded skills (`agent/src/order_triage/skill_loader.py`). When `SKILLS_DIR` is absent (sandbox), `_DOCTRINE` is empty and the catalog renders `(no skills available)` ‚Äî a real branch, not an error.

**6 ¬∑ Model, tools, and the startup coverage gate.** `build_agent` constructs `BedrockModel(model_id=cfg.bedrock_model_id, ‚Ä¶)`. The code default in `agent/src/order_triage/config.py` is `anthropic.claude-opus-4-8`, but `runtime.tf` injects `BEDROCK_MODEL_ID = var.bedrock_model_id` = `amazon.nova-lite-v1:0` at deploy ‚Äî **Nova Lite is what actually runs** (`infra/CLAUDE.md` "Deployed reality"). `additional_args.requestMetadata` tags every Converse with opaque `agent`/`actor`/`session`/`turn` ids, each passed through `_rm_value` to strip `_RM_DISALLOWED` chars (incl. `@`) and truncate to 256 (`agent.py`). Tools come from `get_tools(extra_tools)` (`agent/src/order_triage/tools/__init__.py`): `_local_tools()` = `[search_policies, describe_entity, load_skill]` **plus** the Gateway `extra_tools`. `_assert_action_coverage` is a **hard fail-fast startup gate**: every action any fetched skill can `invoke` must map through `ACTION_IMPLEMENTATIONS` (`{"raiseException": "orders___flagOrder"}`) to a Gateway tool that is actually registered ‚Äî otherwise `build_agent` aborts with `SkillActionCoverageError`. This is the load-bearing invariant that ties the skills corpus to the live Gateway tool surface.

**7‚Äì8 ¬∑ The Strands event loop (model ‚áÑ tools).** `agent.stream_async(prompt)` runs the reasoning loop. Each cycle calls Nova Lite via `ConverseStream`; when **both** `cfg.guardrail_id` and `cfg.guardrail_version` are set (default-on via `var.enable_guardrail` injecting `BEDROCK_GUARDRAIL_ID`/`VERSION` in `runtime.tf`), Strands attaches `guardrailConfig` with `guardrail_stream_processing_mode="async"` and `guardrail_redact_input=False` (`agent.py:guardrail_kwargs`). Note: `agent.py` wires the id/version generically ‚Äî the **PROMPT_ATTACK-only, no-PII** content policy itself lives in `infra/terraform/guardrail.tf` (rationale in ADR-0003: the OBO agent reads customer PII end-to-end by design). The model streams text deltas and `toolUse` blocks back into the loop.

**9‚Äì10 ¬∑ LOCAL tools vs. MCP Gateway tools.** When the model emits a `toolUse`, Strands dispatches to one of two surfaces. **LOCAL tools never traverse the Gateway:**
- `search_policies(query)` ‚Üí `boto3 "bedrock-agent-runtime".retrieve(knowledgeBaseId, numberOfResults=3)` into the Titan-v2 / S3-Vectors Knowledge Base (`agent/src/order_triage/tools/knowledge.py:_kb_retrieve`).
- `describe_entity(api_name)` ‚Üí the ontology reverse index in `bindings.json` + `ontology.compiled.json` (`agent/src/order_triage/tools/ontology.py:OntologyLoader`). Its `EntityView` returns **skills + actions + KB tags + related entities (via `linkTypes`) + properties + datasource** ‚Äî so this tool is itself the ontology‚ÜíKB bridge, not just a skills pointer. Its docstring warns that ontology names are **not** Snowflake runtime field names and must never be passed as tool args.
- `load_skill(name)` ‚Üí the full markdown body of a fetched skill on demand (`agent/src/order_triage/tools/skills.py`).

**MCP Gateway tools** ‚Äî `sap___getCreditStatus`, `orders___flagOrder`, and `snowflake___ask` (the exact set enumerated in `gateway.py`'s module docstring) ‚Äî go over the MCP session to the **AgentCore Gateway (MCP)**, which **Cedar-authorizes every call** (`infra/terraform/policy.tf`: principal `AgentCore::OAuthUser`, guard `principal.hasTag("scp")`, permits `permit_sap_read` / `permit_flag` / `permit_snowflake_ask`) then egresses to the backend Lambdas (SigV4 for sap/orders, OBO `TOKEN_EXCHANGE` for snowflake ‚Äî the data plane, out of scope here). `snowflake___ask` is the single Snowflake analytics tool: a natural-language question that the snowflake Lambda answers via Cortex Analyst over the `ORDERS_SV` semantic view and runs as the signed-in user (ADR-0008). Tool results feed back into the loop, which continues until the model produces the final answer.

**11 ¬∑ Memory ‚Äî short-term + per-user long-term.** When `session_id` is non-null, `build_session_manager` (`agent/src/order_triage/memory.py`) returns an `AgentCoreMemorySessionManager` (`AGENTCORE_MEMORY_ID` from `memory.tf`); when it is null the agent is stateless single-shot (manager is `None`). It persists each turn as short-term events and retrieves extracted long-term memories, injecting them as a `<user_context>` block on the latest user message. `_RETRIEVAL` selects three namespaces ‚Äî `/facts/{actorId}` (SEMANTIC), `/preferences/{actorId}` (USER_PREFERENCE), and `/summaries/{actorId}/{sessionId}` (SUMMARIZATION) ‚Äî all keyed by the request `actor_id`, so long-term memory is partitioned per user (`relevance_score = 0.3`). The strategies are `aws_bedrockagentcore_memory_strategy.{semantic,summary,preferences}` in `memory.tf`. The system prompt explicitly tells the model to treat `<user_context>` as background, **never** as evidence for flagging.

**12‚Äì13 ¬∑ Streaming timeline + token telemetry.** `runtime.py` forwards answer/thinking text bare and runs every other event through `stream_steps.step_events` (`agent/src/order_triage/stream_steps.py`), which classifies Strands `message` events into typed `__step__` timeline entries ‚Äî `tool_call` `{name, input}`, `tool_result` `{status, text}`, `reason` ‚Äî for a Claude-Code-style client timeline. After the stream completes, `_emit_usage_metric` reads `agent.event_loop_metrics.latest_agent_invocation.usage` (the **per-turn** total, not accumulated) and prints one EMF line ‚Äî Namespace `OrderTriage/Agent`, dimensions `[agent_id, model_id]` only, with `session_id` / `actor_id` / `cache_*` as high-cardinality **root** fields (never dimensions). That EMF metric, the Bedrock model-invocation log (PII-masked), and the OTEL `gen_ai` spans (`OTEL_SEMCONV_STABILITY_OPT_IN = gen_ai_latest_experimental,gen_ai_tool_definitions` in `runtime.tf`) feed CloudWatch GenAI Observability. Telemetry never raises ‚Äî a failure must not break the turn.

## Status & caveats

- **Deployed model is Nova Lite (`amazon.nova-lite-v1:0`)** via `BEDROCK_MODEL_ID`. The Python default `anthropic.claude-opus-4-8` in `config.py` is overridden at deploy and is misleading if read in isolation (`runtime.tf`; `agent` CLAUDE.md "Deployed reality").
- **Guardrail is default-on (`enable_guardrail`) but PROMPT_ATTACK input-filter ONLY** ‚Äî no PII policy by design, because the OBO agent reads customer PII end-to-end (ADR-0003). It is also a **silent no-op unless BOTH** `BEDROCK_GUARDRAIL_ID` *and* `‚Ä¶_VERSION` env vars are set (`agent.py` gate); empty env (sandbox) = no guardrail. The PROMPT_ATTACK literal lives in `guardrail.tf`, not in agent code.
- **Skills / ontology / KB docs are FETCHED content** (`make skills`), **gitignored** in the agent repo and pinned to an `knowledge` git tag ‚Äî not source in this repo. The agent **degrades gracefully** to an empty skills catalog / empty ontology when absent: `SYSTEM_PROMPT` renders `(no skills available)`, `_DOCTRINE` is empty, `describe_entity` returns empty (`skill_loader.py`, `ontology.py`).
- **The coverage gate is fail-fast**, not best-effort: a fetched skill whose `invokes` action lacks an `ACTION_IMPLEMENTATIONS` mapping ‚Äî or whose mapped Gateway tool is not registered ‚Äî aborts `build_agent` with `SkillActionCoverageError` (`tools/__init__.py`).
- **Memory is active only when `session_id` is non-null**; `session_id=None` yields a stateless single-shot agent with no persistence (`memory.py`). `retrieval_config relevance_score = 0.3` is a conservative starting floor to tune from traces, not a validated value (ADR-0002).
- **Skill bodies reference abstract verbs** (`ask_orders`, `sap_credit_check`, `score_order`, `flag_order_for_review`) that are skill-authored pseudonyms, **NOT** the live Gateway tool names; only `raiseException ‚Üí orders___flagOrder` is mapped in `ACTION_IMPLEMENTATIONS`. `score_order` is an agent-specific heuristic intentionally **not** promoted into the enterprise knowledge layer (knowledge-repo enterprise-scope).
- **One Snowflake action now: `snowflake___ask`** ‚Äî the four fixed reads (getOrders/getOrder/getCustomer/listCustomers) collapsed into a single Cortex-Analyst-backed analytics tool (`permit_snowflake_ask` in `policy.tf`; `gateway.py` docstring lists `snowflake___ask` + sap + orders). Fine-grained row/column governance moved into the `ORDERS_SV` semantic view + Snowflake RLS, enforced under the user's OBO token (ADR-0008).
- **The snowflake target's OBO `TOKEN_EXCHANGE` credential and the App Signals / online-eval wiring** are applied out-of-band via `terraform_data` + AWS CLI local-exec and have **NO drift detection** (`snowflake_lambda.tf`; ADR-0004/0005). The `api_key` (`X-API-Key`) credential is the creation-time placeholder / SigV4-path fallback.
- **EMF token-metric auto-extraction over the vended-log delivery pipeline is not guaranteed**; a CloudWatch Logs metric filter may be needed (`runtime.py:_emit_usage_metric` NOTE). The structured log line is queryable via Logs Insights regardless.
- **Backend Lambdas, Cedar enforcement detail, OBO brokering, and Snowflake RLS** are collapsed here as one out-of-scope node ‚Äî the full request/identity data plane is `infra/docs/architecture/data-plane.md`.
- **`ontologyVersion` in the fetched `bindings.json` is `0.5.0`** ("Agro-Commodity Supply Chain Ontology", 6 skills incl. `examples/`) while only **4** `*.skill.md` are baked into the agent image; the agent consumes top-level `skills/*.skill.md` only, not `skills/examples/*.skill.yaml` (knowledge CLAUDE.md "Consuming side").
