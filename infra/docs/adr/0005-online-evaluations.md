# ADR-0005: AgentCore online Evaluations (opt-in LLM-judge over sampled traces)

**Status:** Accepted (mechanism) — IaC implemented & deployed 2026-06-23; online-eval-config `order_triage_online_evals-7nK2307vE8` is `ACTIVE`. **Scoring is DEFERRED (2026-06-24):** every evaluation run errors with `AgentSpanMappingException` and produces zero numeric scores. This is now confirmed an **upstream Strands-1.44 ↔ AgentCore-eval delivery/mapper incompatibility**, not an agent-side fixable defect — two agent-side telemetry shims were deployed and live-validated and both still scored zero (see **D7** / Risks **R1**). Evaluations stay **enabled** (harmlessly erroring) and the gen_ai content events are captured; the *mechanism* decision (native, opt-in, CLI-provisioned) stands. The *quality-score output* is deferred pending an AWS-side fix.
**Date:** 2026-06-23 (mechanism); 2026-06-24 (scoring deferred)
**Deciders:** Anil Choudhary (proposer); platform + security owners
**Related:** [ADR-0001](0001-user-impersonation-obo.md) (the CUSTOM_JWT inbound identity these traces carry), [ADR-0002](0002-agentcore-memory-activation.md) (the same per-turn gen_ai spans + ADOT auto-instrumentation this consumes), [ADR-0004](0004-observability-finops.md) (Transaction Search → `aws/spans`, the data source this depends on); the observability spike — `../research/observability-spike.md` §2.3/§9, `../research/observability-phase3-research.md` §2.3/§5-D7/§6, and the `observability-spike` memory note.

## Context

Phase-3 observability (`../research/observability-phase3-research.md`) defines seven audience-scoped dashboards. **Dashboard 7 — Feedback & Evaluation** ("is answer quality holding, did a model swap regress it?") is the only one with **no native data source flowing today**: the stack emits volume / latency / error / token / trace signals but **no quality, groundedness, or tool-trajectory scores** (research §3, "Eval/quality/groundedness scores — native + feasible, not wired"). The research benchmark (§1, row 4) marks online + offline LLM-as-judge evaluation as table-stakes for a best-in-class GenAI observability surface.

Three forces shape the decision:

1. **A native evaluator already exists and consumes what we already emit.** AgentCore Evaluations (GA 2026-03-31) runs LLM-judge evaluators **online over sampled live traces** and publishes scores parented to the originating span (research §2.3, spike §63). Critically it reads the **gen_ai spans the ADOT auto-instrumentation already produces** (ADR-0002 D3, ADR-0004 Transaction Search → `aws/spans`) — so the only thing missing is the evaluation *config*, not new agent instrumentation. The spike is explicit that "eval is third-party-only" is the **old, wrong** thesis (spike §63).

2. **An LLM judge has a real per-trace cost.** Each sampled trace is scored by invoking a judge model (here at 100% sampling). That is recurring spend proportional to traffic, unlike the zero-marginal-cost vended metrics. For a demo that is mostly idle, a judge that fires on every trace is wasteful if left on by default.

3. **There is no first-class Terraform resource for the config.** Neither `hashicorp/aws` nor `awscc` exposes a `*_online_evaluation_config` resource; the control-plane API `bedrock-agentcore-control create-online-evaluation-config` is the only way to create it. This is the same situation as the Application Signals SLOs (research §9-3f) and the account X-Ray routing (`observability.tf`), where we already fall back to `terraform_data` + local-exec.

Implementation lives in `terraform/modules/observability/evaluations.tf`, gated by `var.enable_online_evaluations` (`terraform/variables.tf:178`, default `false`; module copy `terraform/modules/observability/variables.tf:38`). The Feedback dashboard (`terraform/modules/observability/dashboards.tf:461`) reads its banner from the same flag.

## Decision

**D1 — Use the native AgentCore online Evaluations, not a custom harness or a third party.** Evaluation is provisioned through `bedrock-agentcore-control create-online-evaluation-config` against the spans already in `aws/spans`. We do **not** build a periodic Lambda/cron LLM-judge harness, and we do **not** adopt Langfuse/Phoenix/Ragas/Braintrust. Rationale: the native service consumes the exact gen_ai spans we emit (zero new agent code), keeps quality signal inside the same CloudWatch plane as every other dashboard, and stays inside the stack's hard "100% AWS-native, no Langfuse/Arize/Datadog" constraint (research §0 hard constraint, §10). Because the wire format remains OTEL gen_ai, an OTEL-fed third party is still a drop-in *later* if a genuine native gap (annotation queues, dataset curation) ever comes into scope (spike §111-112) — adopting native now sacrifices nothing.

**D2 — Opt-in, default OFF.** `var.enable_online_evaluations` defaults to `false`. Every evaluation resource is `count = var.enable_online_evaluations ? 1 : 0` (the exec role, its policy, and the `terraform_data`). The deliberate reason is cost: the judge runs **per sampled trace** at `samplingPercentage = 100` (`evaluations.tf:12`), i.e. real recurring model spend that scales with traffic — unacceptable as a silent default for an idle demo. When off, nothing is created and the flag only flips the Feedback dashboard banner (variable description, `variables.tf:178`; banner in `dashboards.tf:474`). Enablement is a reviewed operator action (`ENABLE_ONLINE_EVALUATIONS=true` in `.env`), not an always-on platform default.

**D3 — Provision out-of-band via `terraform_data` + the control-plane CLI, with retry on IAM eventual-consistency.** Because no native TF resource exists (see Context force 3), `resource "terraform_data" "online_evaluations"` (`evaluations.tf:65`) runs a `local-exec` calling `aws bedrock-agentcore-control create-online-evaluation-config --cli-input-json '${local.online_eval_payload}'`. `triggers_replace = [local.online_eval_payload]` re-runs only when the payload changes; a `when = destroy` provisioner looks the config id up by name and deletes it (idempotent — `[ -n "$id" ] && [ "$id" != "None" ]`). The create loop retries **6× with 15s sleeps, but only on the `does not have permissions` error** — the exec role and its inline policy are created in the *same apply* and IAM is eventually consistent, so the service can evaluate the role before the grant propagates; **any other error fails fast** (`grep -q "does not have permissions" || exit 1`). This propagation race was confirmed real in practice (PR #102) — the retry, not a broader `logs:CreateLogGroup → *` grant, is what landed it (PR chain #99→#102, HANDOVER).

**D4 — A dedicated evaluations-exec IAM role with least-privilege.** `aws_iam_role.evaluations` (`evaluations.tf:35`, name `${var.name_prefix}-evaluations-exec` = `order-triage-evaluations-exec`) is assumable only by `bedrock-agentcore.amazonaws.com` and confined by a `aws:SourceAccount == var.account_id` condition. Its inline policy (`evaluations.tf:49`) grants exactly four capabilities: **ReadTraces** (`logs:StartQuery`/`GetQueryResults`/`GetLogEvents`/`FilterLogEvents`/`DescribeLogGroups`) to read the trace + content groups; **WriteResults** (`logs:CreateLogGroup`/`CreateLogStream`/`PutLogEvents`/`PutRetentionPolicy`/`DescribeLogStreams`) scoped to the results namespace `/aws/bedrock-agentcore/evaluations/*`; **InvokeJudge** (`bedrock:InvokeModel`) for the judge call; **PublishScores** (`cloudwatch:PutMetricData`). The `arn` is wired into the config payload as `evaluationExecutionRoleArn` (`evaluations.tf:28`).

**D5 — Five built-in evaluators over the agent's own service spans.** The config (`local.online_eval_payload`, `evaluations.tf:9`) names `Builtin.Correctness`, `Builtin.Helpfulness`, `Builtin.Faithfulness`, `Builtin.ToolSelectionAccuracy`, and `Builtin.ToolParameterAccuracy` — the quality + tool-trajectory pair that Dashboard 7 wants (research §5-D7). `dataSourceConfig.cloudWatchLogs` reads **two groups**: `aws/spans` (the trace spans, from ADR-0004 Transaction Search) **and** `local.runtime_default_log_group` (`/aws/bedrock-agentcore/runtimes/${runtime_id}-DEFAULT`, which carries the per-span gen_ai content/message events the judge correlates by `spanId`), scoped to `serviceNames = ["order_triage.DEFAULT"]`. The config name `order_triage_online_evals` is underscored deliberately — it must match `[a-zA-Z][a-zA-Z0-9_]{0,47}` with no hyphens (`evaluations.tf:10`; this was PR #100's lesson).

**D6 — Results consumption (honest, partly-unconfirmed).** The design intent (research §5-D1/§5-D7, `dashboards.tf:484`) is that scores publish to the `Bedrock-AgentCore-Evaluations` CloudWatch namespace, read by the Feedback dashboard's `SELECT AVG(Score) ... GROUP BY EvaluatorName` Metrics-Insights tile. **Live observation contradicts this:** evaluation output currently lands as OTEL **log records** (`gen_ai.evaluation.result`) in the results log group `/aws/bedrock-agentcore/evaluations/results/<config-id>`, not (yet observed) as a metric in that namespace. The dashboard's metric-namespace assumption is therefore **unverified**; rebuilding the Feedback widget as a Logs Insights widget over the results group is a tracked follow-up ("Fix B", HANDOVER). This is recorded here so the namespace claim is not mistaken for validated fact.

**D7 — SCORING is DEFERRED as an upstream incompatibility; no further agent-side reshaping (2026-06-24).** The `AgentSpanMappingException` (R1) is an **AWS-side** delivery/mapper defect, not an agent telemetry bug we can patch. Two shims in `order-triage-agent` were deployed and live-validated against config `order_triage_online_evals-7nK2307vE8`, both producing **zero scores**:
- patching `Tracer._add_event` (reshaping the `{role, parts}` envelope toward native `{role, content}`) — AgentCore **re-wrapped** the delivered content, so the eval mapper still failed to parse `user_query`;
- patching `Tracer._map_content_blocks_to_otel_parts` (emit native Bedrock ContentBlocks *as* the OTel parts) — the **system** message then arrived native, but AgentCore delivers the **user** message's content as a whole serialized `[{role, parts}]` aggregate the mapper cannot parse.

The conclusion: AgentCore's delivery of Strands 1.44's `gen_ai_latest_experimental` `gen_ai.input.messages` aggregate is **inconsistent** (per-message native for the system message, whole-aggregate-wrapped for the user message), and its eval mapper cannot round-trip it. No single agent-side content shape satisfies both the `user_query` parse (wants plain/flat content) **and** the tool-trajectory evaluators (want structured tool detail). Both shims were therefore **reverted** in `order-triage-agent` (the `telemetry_eval_compat.py` monkeypatch deleted, the temporary `strands-agents>=1.44.0,<1.45.0` pin relaxed back to `>=1.40.0`): the shim was a no-op for evals and it made the OTel spans non-standard (native blocks where OTel parts are expected), which can confuse the GenAI-Observability content display. We **keep** Evaluations enabled (harmlessly erroring) and **keep** `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` on the runtime (`runtime.tf`) — gen_ai content capture has standalone GenAI-Observability value independent of scoring. Revisit conditions are in R1 / Action items.

## Options considered

**Eval mechanism (D1).**

| Option | Assessment | Verdict |
|---|---|---|
| **Native AgentCore online Evaluations** | GA, consumes the gen_ai spans already in `aws/spans` (zero new agent instrumentation), scores parent to the source span, stays in-account and inside the no-3p constraint | **Chosen** |
| Custom periodic LLM-judge harness (Lambda/cron reading `aws/spans`, calling `bedrock:InvokeModel`, emitting EMF scores) | Full control over rubric, but re-implements span parsing + judge orchestration + score publishing that the native service already does; ongoing code to own; no parenting-to-span in the console | Rejected |
| Third-party eval platform (Langfuse / Arize Phoenix / Ragas / Braintrust) | Richer eval UX + annotation queues + dataset/experiment diff, but violates the hard "no Langfuse/Arize/Datadog" constraint (research §0); adds PII-egress of order/credit/dispute content; unnecessary given a native option exists | Rejected (deferred as OTEL-additive only if a true native gap appears — spike §111-112) |

**Default posture (D2).**

| Option | Assessment | Verdict |
|---|---|---|
| **Default OFF (opt-in)** | A judge model runs per sampled trace (100% sampling) = recurring spend scaling with traffic; wasteful on an idle demo; enablement is a reviewed, costed decision | **Chosen** |
| Default ON | Best out-of-box quality signal, but silently incurs per-trace judge cost the moment any traffic flows; surprising for anyone applying the stack | Rejected |

**Provisioning (D3).** Native TF resource (none exists in `hashicorp/aws` or `awscc` — not an option) vs. **`terraform_data` + control-plane CLI** (chosen — the established idiom in this module for resources without a provider, e.g. the SLOs §9-3f and the X-Ray account routing in `observability.tf`; cost is no drift detection and imperative create/destroy) vs. a fully manual console/CLI step outside Terraform (rejected — un-versioned, no destroy cleanup, no IAM-propagation retry). The retry-only-on-`does not have permissions` design (rather than blanket retry) keeps genuine config errors failing fast.

## Consequences

**Becomes easier**
- Dashboard 7 has a native quality source for the first time: Correctness / Helpfulness / Faithfulness + tool-selection/parameter accuracy on real traffic, with **no new agent code** — the same gen_ai spans from ADR-0002/ADR-0004 are simply read by the judge.
- Enabling/disabling quality scoring is a single boolean (`enable_online_evaluations`) with full create + destroy lifecycle handled by the `terraform_data` provisioners; the dashboard banner self-documents the current state.
- Stays 100% AWS-native; no new data-egress surface, no vendor account, no second telemetry pipeline.

**Becomes harder / new burden**
- **Recurring judge cost** whenever enabled — a model invocation per sampled trace at 100% sampling. Must be budgeted; the sampling rate is the lever to turn down.
- **Verbatim content in two log groups.** Faithfulness/correctness judging requires the prompt/response content, which includes Snowflake/SAP order, credit, and dispute data — now logged unmasked in the runtime `-DEFAULT` group and read by the eval role (per the operator's "apply now, no hardening" sign-off, HANDOVER). Data-classification of these groups must be revisited before real users.
- **An out-of-band resource Terraform cannot fully see.** Drift (e.g. someone deletes the config in the console) is invisible to `plan`; only the payload hash triggers re-create.
- **A results-shape assumption to reconcile.** The Feedback dashboard reads a metric namespace that has not been observed to populate; the real output is log records (D6). Carrying both until Dashboard 7 is rebuilt.

**Will need to revisit**
- **The `AgentSpanMappingException` blocker (R1) — deferred upstream.** Evaluations run and *error* rather than score (enabled but producing nothing). Revisit when AWS's eval mapper supports the Strands-1.44 `parts`/aggregate shape, or evaluate the untested `aws-opentelemetry-distro` → `telemetry.auto.version 0.12.2-aws` downgrade (affects all observability, weigh carefully). File the incompatibility as AWS feedback.
- **Sampling rate.** 100% is fine while idle/cheap; lower it (e.g. `samplingPercentage`) once traffic and cost are real.
- **Dashboard 7 widget vs results shape (D6).** Rebuild as Logs Insights over `/aws/bedrock-agentcore/evaluations/results/*` (parse `gen_ai.evaluation.name` + score, filter `error != 1`) once scores actually exist, or confirm the metric namespace does populate.
- **Evaluator set.** Five Builtins today; add deterministic trajectory matchers or safety/harmfulness evaluators if the rubric grows.

## Risks

1. **Version skew blocks scoring — confirmed upstream (RESOLVED → DEFERRED, 2026-06-24).** With `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` (set on the runtime to make the aggregate `body.input.messages` content event exist at all), Strands 1.44 emits message `content` in the **OTel-v1.37 `parts` shape** (`'[{"type":"text","content":"…"}]'`), which the AgentCore eval parser cannot read → `AgentSpanMappingException: Failed to parse user_query from agent-span` on all 5 evaluators, **0 numeric scores**. Dropping `gen_ai_latest_experimental` is not viable — it re-triggers the prior `LogEventMissingException` (confirmed live on the earlier config `dJ3KO96WVM`). **Both agent-side reshaping options were tried and failed (see D7):** two shims (`Tracer._add_event`, then `Tracer._map_content_blocks_to_otel_parts`) were deployed and live-validated and both still scored zero, because AgentCore **re-wraps** the delivered content (the user message arrives as a whole serialized `[{role, parts}]` aggregate). *Resolution:* treat as an upstream Strands-1.44 ↔ AgentCore-eval-mapper incompatibility — keep evals enabled (harmlessly erroring), revert the shim, **file with AWS**, and revisit when **(a)** the AgentCore eval mapper supports the v1.37 `parts`/aggregate shape (the standard direction), or **(b)** — *untested, weigh carefully* — pinning `aws-opentelemetry-distro` toward `telemetry.auto.version 0.12.2-aws` (the version in AWS's working eval doc example; live is `0.18.0`) changes the delivered shape. Option (b) affects **all** observability instrumentation, not just evals, so it is not a casual lever.
2. **Per-trace judge cost runs away under load.** 100% sampling × an LLM call per trace is unbounded by traffic. *Mitigation:* default OFF (D2); lower `samplingPercentage` before enabling on anything but a low-traffic demo; pair with a cost-anomaly alarm.
3. **Sensitive content readable by the eval role + persisted in logs.** *Mitigation:* exec role read-scope is least-privilege (D4); data-protection masking on the runtime/results groups and a real data-classification review are required before onboarding real users (currently deferred per sign-off).
4. **IAM eventual-consistency on a cold apply** causes a spurious create failure. *Mitigation:* the 6×15s retry-only-on-`does not have permissions` loop (D3), validated as the fix in PR #102; any non-permission error still fails fast.
5. **Dashboard reads the wrong results surface (D6).** The `Bedrock-AgentCore-Evaluations` metric tile may stay empty even once scoring works. *Mitigation:* tracked "Fix B" to rebuild Dashboard 7 over the results log group; banner already states "populates after enablement" so it does not falsely imply data.
6. **Out-of-band config drifts from state.** *Mitigation:* `triggers_replace` on the payload hash re-creates on change; the `when=destroy` provisioner cleans up on disable; periodically reconcile with `list-online-evaluation-configs`.
