# ADR-0004: AWS-native observability & FinOps (zero third-party)

**Status:** Accepted — implemented & deployed 2026-06-23 (Phases 0–4 LIVE + validated; refactor into `terraform/modules/observability/` deployed via PR #97). Operational telemetry (delivery, EMF token metric, model-invocation logging behind the PII mask, 7 dashboards, alarms + composite, 2 SLOs, 2 Contributor Insights rules) is live and validated by a `make status` invoke. Two known-incomplete edges: the `OrderTriage/Agent` EMF-auto-extraction-vs-metric-filter path is validated working live (no filter needed), and online LLM-judge **scoring** is out of scope here (deferred to ADR-0005 — config is ACTIVE but blocked on `AgentSpanMappingException`).
**Date:** 2026-06-23
**Deciders:** Anil Choudhary (proposer); platform + security owners
**Related:** [ADR-0001](0001-user-impersonation-obo.md) (CUSTOM_JWT inbound + per-user OBO — the identity these signals attribute against), [ADR-0002](0002-agentcore-memory-activation.md) (whose D3 already wired the *memory* slice of log/trace delivery — this ADR is the platform-wide superset that generalizes that exact pattern), [ADR-0003](0003-bedrock-guardrail.md) (the guardrail's deliberate **no-PII-policy** choice — this ADR is where the complementary PII control lives, as a CloudWatch Logs data-protection mask), [ADR-0005](0005-online-evaluations.md) (online LLM-judge Evaluations — referenced, not decided here); spike docs `../research/observability-spike.md`, `../playbooks/observability-impl-plan.md`, `../research/finops-spike.md`.

## Context

The stack was **already substantially AgentCore-native** before this work: the runtime image launches under `opentelemetry-instrument` with `aws-opentelemetry-distro` (agent `Dockerfile`), vended `AWS/Bedrock-AgentCore` metrics auto-emit, account-level CloudWatch Transaction Search was wired, and ADR-0002 had wired log/trace delivery for the **Memory** resource. The observability spike's central finding was therefore **not** a native-vs-third-party choice — it was a **deeply under-used native surface**: no dashboards, no alarms, no SLOs, no Bedrock model-invocation logging, no per-resource Tracing/log delivery on Runtime or Gateway, and the per-turn Strands token-usage event was being discarded in the runtime entrypoint (the agent's `runtime.py`, before `emit_usage_metric` was wired in).

A live-console correction on 2026-06-23 sharpened the gap: the AgentCore console showed **Runtime and Identity Tracing "Not enabled" / 0 log deliveries**. Terraform had wired delivery for the Memory resource only (ADR-0002 D3), so **only the vended metrics flowed** — the `InvokeAgentRuntime`/gen_ai/token spans were never delivered to CloudWatch. The account-level Transaction Search routing being on does **not** by itself deliver a resource's traces; the per-resource Tracing toggle is the gate.

The FinOps spike established the cost picture and its hard limits. The deployed model is **`amazon.nova-lite-v1:0`** (`runtime.tf` `BEDROCK_MODEL_ID` ← `variables.tf:14` default; the Python Opus default in the agent's `agent.py` is overridden at deploy and does not run). AWS billing alone stops at account → service → resource/IAM-principal → tag, and the agent invokes Bedrock under **one shared runtime execution role**, so CUR/Cost Explorer collapse every Entra user into that one principal. The lower three of the five attribution levels — **actor, session, message** — are **structurally not derivable from billing** and must be self-instrumented. The spike's recommended posture: **Tier-1 AWS-native + `requestMetadata`**, with token→dollar always treated as an estimate (CUR is the only invoice-grade source).

Two forces shaped every decision: **stay 100% AWS-native / zero third-party / all telemetry in-account** (no PII egress — important given ADR-0003's no-PII guardrail), and **never let a body land unmasked** (any record written before a CloudWatch Logs data-protection policy exists is stored unmasked permanently and cannot be masked retroactively).

The implementation also carried a **structural refactor (PR #97)**: the ~56 observability resources were extracted from a top-level `observability.tf` into `terraform/modules/observability/` (`observability.tf` = log/trace delivery, `invocation_logging.tf`, `log_groups.tf`, `monitoring.tf`, `dashboards.tf`, `alarms.tf`, `slo.tf`, `queries.tf`, `evaluations.tf`); the root `terraform/observability.tf` is now a single module block. Migration used a one-time `terraform state mv` (not `moved{}` blocks — rejected as provenance-in-code), moving 56 resources with 0 destroy.

## Decision

**Posture: AWS-native GenAI Observability with ZERO third-party tooling.** Every signal stays in-account (CloudWatch / X-Ray-Transaction Search / GenAI Observability / Application Signals). LLM-quality-workflow features that genuinely have no native equivalent (multi-provider judge choice, annotation queues, per-trace dollarization) are all **OTEL-additive** later via a second OTLP exporter, so this choice sacrifices nothing reachable. AgentCore Evaluations is itself native (see ADR-0005).

**D1 — Per-resource log + trace delivery for Runtime, Gateway, and KB; Identity transitively.** Generalize ADR-0002's Memory delivery trio (`aws_cloudwatch_log_delivery_source` → `_destination` → `_delivery`) to every resource, in `modules/observability/observability.tf`:
- **Runtime**: `APPLICATION_LOGS` and `USAGE_LOGS` → vended CloudWatch Logs groups (`/aws/vendedlogs/bedrock-agentcore/runtime/{APPLICATION_LOGS,USAGE_LOGS}/${runtime_id}`); `TRACES` → X-Ray. `USAGE_LOGS` is the only native per-session 1-sec vCPU/GB-hours signal; the provider accepts the enum (no local-exec fallback was needed — validated).
- **Gateway**: `APPLICATION_LOGS` → CWL; `TRACES` → X-Ray (the two-span Call-Tool SERVER+CLIENT structure with `TargetExecutionTime`, plus the OBO `GetWorkloadAccessTokenForJWT(issuer, user_sub)` spans — the highest-value audit surface given ADR-0001's per-user OBO value-prop).
- **Knowledge Base**: `APPLICATION_LOGS` only (the sole `log_type` a KB supports; source is the KB ARN, not the data-source ARN), plus a `KBIngestionJobFailed` metric filter (no native ingestion metric exists).
- **Identity / WorkloadIdentity** has **no standalone resource** — it lights up transitively via the above (console-confirmed Enabled). Wiring a delivery for it would be a guaranteed-fail apply, so it is a deliberate non-action.

X-Ray `TRACES` deliveries take `delivery_destination_type = "XRAY"` with no config block and `depends_on = [terraform_data.transaction_search]`; the existing `aws_cloudwatch_log_resource_policy.xray_spans` already grants xray→`aws/spans` `PutLogEvents`, so no new resource policy. `AGENT_OBSERVABILITY_ENABLED` was confirmed **not** required — the hosted runtime auto-instruments; 37 gen_ai spans with `gen_ai.usage.*` landed in `aws/spans` after a live invoke without it. Transaction Search head-indexing is dialed by `aws_xray_indexing_rule "Default"` ← `var.trace_indexing_percentage` (default 100; cost/search lever only — it does **not** reduce span storage).

**D2 — Bedrock model-invocation logging, gated behind a CloudWatch Logs PII mask — THIS is where PII is masked.** `modules/observability/invocation_logging.tf` enables the account+region-singleton `aws_bedrock_model_invocation_logging_configuration` → a dedicated CWL group (`/aws/bedrock/${name_prefix}/modelinvocations`) with a Bedrock-trusted role scoped by `SourceAccount` + `SourceArn`. This is the canonical AWS-native per-**call** token-count + verbatim prompt/completion source (and it captures the KB Titan-embed and Memory-summarization InvokeModels under their own principals). It is the **deliberate complement to ADR-0003**: the guardrail intentionally carries no PII policy, so PII is masked **here, at the log layer**, by `aws_cloudwatch_log_data_protection_policy.bedrock_invocations` — two statements (Audit then Deidentify) over an **identical** array of 5 verified managed identifiers: `EmailAddress`, `PhoneNumber-US`, `Ssn-US`, `DriversLicense-US`, `CreditCardNumber`. Generic `Name`/`Address` are **not** valid managed identifiers (they would reject the whole policy) and free-text customer names have no managed identifier — accepted as unmasked. The mask resource is a **hard `depends_on`** of the logging config (non-reversible ordering — see Risks). `large_data_delivery_s3_config` is deliberately **unset**: bodies >100KB truncate in CWL rather than spilling to a Bedrock-managed S3 path the CWL mask cannot cover.

**D3 — Per-turn EMF token metric + Converse `requestMetadata` for the lower attribution levels.** Token usage was being discarded; the agent now captures it without third-party tooling:
- `agent_kit.infra.metrics::emit_usage_metric` reads `agent.event_loop_metrics.latest_agent_invocation.usage` (the per-turn total — `accumulated_usage` is never zeroed and would over-count on a reused Agent) after the stream completes and prints one **EMF document** to stdout in namespace `OrderTriage/Agent` (the agent passes its `METRIC_NAMESPACE`). **Cardinality rule (load-bearing):** the only metric **dimensions** are `agent_id` + `model_id`; `session_id`, `actor_id`, and `cache_*` are root **log fields** — high-cardinality keys as dimensions would explode the custom-metric bill. The Strands tracer already sets `gen_ai.usage.*` on its own spans, so **no span attribute is added** — the EMF metric is the genuine gap (CloudWatch otherwise has no token metric).
- The agent's `build_agent` injects `additional_args={"requestMetadata": {agent, actor, session, turn}}` (from `kit.request_metadata(...)`) on `BedrockModel`. Strands spreads `additional_args` at the Converse **top level** (the `requestMetadata` param — not `additionalModelRequestFields`). Values are sanitized to opaque ids only (`agent_kit.prompt:_rm_value`: charset-limited, ≤256 chars, no `@`, empties dropped) so a UPN-shaped subject can never reach the model-invocation log and an odd id can never make Converse reject the turn. The N per-cycle records of one turn share a `turn` uuid → per-message rollup. Dollarization is **not** done here — counts are captured PII-safe and observable; the price map is a dashboard estimate (D6) per FINOPS-SPIKE.

**D4 — Seven audience-scoped dashboards on one drill spine** (`modules/observability/dashboards.tf`): **Operations** (engineers — RED golden signals, downstream App-Signals latency/faults, per-tool `strands.*`, token volume, vCPU/GB-hours), **Incident** (SRE — alarm widget + runbook deep-links to X-Ray Trace Map / Transaction Search / GenAI Observability), **Exec** (leadership — success rate, p99, est. spend, guardrail interventions, health), **FinOps** (token volume + estimated $ + Contributor-Insights top actors/sessions + hidden Memory `LongTermMemoryProcessing` tokens), **Security** (guardrail interventions, Cedar `AllowDecisions` by tool, OBO success/failure, caller-identity audit), **Governance** (per-turn model-invocation audit table, Cedar decisions, KB access latency; banner documents the PII mask), **Feedback** (eval-score trend — populates only after ADR-0005's scoring works; human-feedback row is an explicit gap). Namespace literals were pinned from a live `list-metrics` read-back: vended = `AWS/Bedrock-AgentCore` (hyphen), OTEL app metrics = `bedrock-agentcore`, `ApplicationSignals`, `OrderTriage/Agent`; runtime dims = `[Name, Operation, Resource]`.

**D5 — Alarms → composite → SNS; 2 Contributor Insights rules** (`alarms.tf`, `monitoring.tf`, `queries.tf`): static fault alarms on runtime `SystemErrors`/`Throttles`, App-Signals `Fault` (rolls up any downstream — model/KB/Memory/SAP/Snowflake), and OBO `ResourceAccessTokenFetchFailures` (per-user Snowflake access broken); a **token-usage anomaly** alarm via `ANOMALY_DETECTION_BAND(m1, 2)` on `OrderTriage/Agent` `TotalTokens`; all rolled into a composite **`agent_unhealthy`** (rollup test passed via `set-alarm-state`) → an SNS topic with an `alert_email` subscription (confirmed). KB-ingestion-FAILED alarm is deliberately **outside** the composite (data-freshness, not a live-path fault). `treat_missing_data = "notBreaching"` throughout (the demo idles). Two Contributor Insights rules rank top actors and top sessions by `$.TotalTokens` straight from the runtime `-DEFAULT` log group's JSON fields.

**D6 — FinOps spend is an explicit ESTIMATE, not billing-accurate.** The Exec/FinOps dashboards multiply the EMF token metric by `var.model_input_usd_per_million` / `var.model_output_usd_per_million` (Nova Lite defaults 0.06 / 0.24) via a CloudWatch math expression. The FinOps banner states plainly that token×rate is an estimate and invoice-accurate per-actor $ comes from CUR 2.0 IAM-principal attribution (FINOPS-SPIKE). Retention is bounded by `var.{memory,bedrock_invocation,function}_log_retention_days` (30d defaults). Cost-allocation tags beyond `Project`+`ManagedBy` (FinOps P2.7) were **deferred** — they need org-supplied values + Billing-console activation by the management account.

**D7 — 2 Application Signals SLOs created via `terraform_data` local-exec (a deliberate trade-off).** `slo.tf` defines a latency SLO (p99 ≤ 5s, 99% / rolling 7d) and an availability SLO (system-error-free, 99% / rolling 7d) using the `SliConfig.SliMetricConfig` metric-query form against the vended `Latency`/`SystemErrors` metrics. They are created by `aws application-signals create-service-level-objective ... || update-...` in a local-exec, **not** a native resource, because: `hashicorp/aws` has **no** `aws_applicationsignals_service_level_objective` (issue #39555 open), and `awscc_applicationsignals_service_level_objective` was **broken** — its Cloud Control create-then-stabilize step re-issues the create within a single apply and self-collides with `AlreadyExists`, leaving get-able-but-non-listable zombie SLOs that had to be manually deleted/recreated. The local-exec is imperative (no drift detection, needs the deploy role's `application-signals:*` + `cloudwatch:GetMetricData`) but it is the only path that actually converges. Same local-exec idiom is reused for online evaluations (ADR-0005).

## Options considered

**Tooling posture**

| Option | Assessment | Verdict |
|---|---|---|
| AWS-native only (CloudWatch / X-Ray / Transaction Search / GenAI Obs / App Signals) | Comprehensiveness-complete for operational telemetry; all data in-account (no PII egress, important given ADR-0003); $0 software | **Chosen** |
| Add Langfuse/Phoenix/Braintrust now | Their unique value (multi-provider judges, annotation queues, per-trace dollars) is **OTEL-additive** — a second OTLP exporter alongside the ADOT one, addable later with zero native loss | Deferred |
| LiteLLM proxy | Managed runtime calls the model **inside the container** — no interceptable hop; would require rewriting the agent, relocating creds, and a Postgres/Redis service, and would still be blind to Gateway/Snowflake/KB/Memory | Rejected |
| Datadog/APM | Only if the org already owns it; adds SaaS cost + PII-egress risk (no PII guardrail per ADR-0003); truncates high-cardinality cost tags | Rejected for this stack |

**Token-usage capture mechanism (D3)**

| Option | Assessment | Verdict |
|---|---|---|
| Per-turn EMF metric (`agent_id`+`model_id` dims only) + `requestMetadata` log fields | Cheap, alarmable, queryable; respects the cardinality rule; no span pollution (tracer already sets `gen_ai.usage.*`) | **Chosen** |
| Add a custom span attribute for usage | Duplicates the Strands tracer's existing `gen_ai.usage.*` span attrs | Rejected |
| `session_id`/`actor_id` as EMF dimensions | One metric stream per unique value → custom-metric bill explosion | Rejected (kept as log fields) |
| Derive lower levels from billing | CUR collapses all users into the one shared runtime role — structurally impossible | Rejected (must self-instrument) |

**PII masking site (D2, complement to ADR-0003)**

| Option | Assessment | Verdict |
|---|---|---|
| CloudWatch Logs data-protection mask on the invocation log group | Masks at the log layer (5 verified managed identifiers); keeps full bodies for token/audit value while de-identifying PII; in-account | **Chosen** |
| A PII guardrail policy on the model | ADR-0003 deliberately scoped the guardrail to PROMPT_ATTACK only; PII control belongs at the log sink | Rejected (ADR-0003 decision) |
| Spill >100KB bodies to S3 (`large_data_delivery_s3_config`) | The CWL mask does **not** cover S3; Snowflake rows can exceed 100KB → unmasked-PII surface | Rejected (left unset; bodies truncate) |
| Include `Name`/`Address` identifiers | Not valid CloudWatch Logs managed identifiers — would reject the entire policy | Rejected (excluded) |

**Application Signals SLOs (D7)** — `hashicorp/aws` resource (does not exist, #39555) vs `awscc` resource (**broken** — create-stabilize self-collides) vs `terraform_data` + AWS CLI local-exec (**chosen** — imperative and drift-blind, but the only form that converges; reuses the existing local-exec idiom). The P3.3 latency-p99 alarm already covers the operational need, so the SLOs add burn-rate semantics, not basic alerting.

## Consequences

**Becomes easier**
- Full operational telemetry — trace/span capture (gen_ai + tool + Gateway Call-Tool + OBO + KB Retrieve), structured app logs, vended + EMF metrics, per-session USAGE_LOGS, retention, and PII governance — all native, all in-account.
- The five-level token/cost view from FINOPS-SPIKE: overall/agent from vended metrics + CUR; actor/session/message from `requestMetadata` + the EMF log fields, grouped by `turn`; top-spender leaderboards from Contributor Insights.
- Per-user accountability and authz audit consistent with ADR-0001: OBO `GetWorkloadAccessTokenForJWT(issuer, user_sub)` spans, Cedar `AllowDecisions` by tool, and a caller-identity table — the closest native proxy to the retired "served-as" identity panel.
- Incident response: a single composite `agent_unhealthy` signal + SNS, and seven role-scoped dashboards that all drill down the same `session.id` spine.

**Becomes harder / new burden**
- **PII floor is a hard ordering constraint.** The mask must exist before the logging config writes its first record — `depends_on` enforces it; the apply must never be split such that logging lands first.
- **Telemetry cost to govern.** Transaction Search defaults to 100% indexing; `var.trace_indexing_percentage` dials it. Verbatim invocation logs grow per Converse call; retention vars bound them. Anomaly detectors bill ~$0.30/detector/mo (2).
- **The SLOs are imperative.** `terraform_data` local-exec has no drift detection and needs extra deploy-role perms; verify SLOs via `get-service-level-objective --id`, not `list` (list omits metric-query SLOs).
- **More log groups + deliveries** (runtime ×2, gateway, KB, 3 Lambda, invocation log) to own and budget retention for.

**Will need to revisit**
- **Cost-allocation tags (FinOps P2.7).** `default_tags` is still only `Project`+`ManagedBy`; add `CostCenter`/`Environment`/`team` once the org supplies values and the management account activates the keys in Billing.
- **`OrderTriage/Agent` EMF extraction via the delivery pipeline.** Auto-extraction is validated working live for the runtime `-DEFAULT` group; if a future runtime/log-routing change breaks it, fall back to a CloudWatch Logs metric filter over the same JSON.
- **Model swap economics.** The estimate price map is keyed to `amazon.nova-lite-v1:0` (+ `amazon.titan-embed-text-v2:0`); a swap to an Anthropic tier requires updating the rate vars and an Application Inference Profile for tag-level $ (FinOps).
- **KB ingestion metric-filter JSON path** is provisional until confirmed against one real emitted record.

## Risks

1. **Unmasked-PII window.** Any record written before the data-protection policy exists is stored unmasked permanently. *Mitigation:* the `depends_on` from the logging config to the mask policy; never split the apply; `large_data_delivery_s3_config` left unset; short retention; and the broader data-governance posture is the user's explicit "apply now, no hardening" sign-off — revisit before real customers.
2. **Cost-tag dimension explosion.** Putting `session_id`/`actor_id` in EMF dimensions spawns a metric stream per value. *Mitigation:* the cardinality rule — only `agent_id`+`model_id` are dimensions; everything high-cardinality is a log field (enforced in `agent_kit.infra.metrics` and the alarm/dashboard dims).
3. **Telemetry spend creep.** 100% trace indexing + verbatim logs. *Mitigation:* `var.trace_indexing_percentage`, retention vars, only 2 anomaly detectors.
4. **SLO drift / convergence.** The local-exec is drift-blind and the awscc resource self-collides. *Mitigation:* `terraform_data` create-or-update with a destroy provisioner; documented `get --id` verification; deploy-role `application-signals:*` granted.
5. **Pinned-literal fragility.** Dashboard/alarm/SLO correctness depends on the live-pinned namespace + dimension literals (vended ns is the hyphenated `AWS/Bedrock-AgentCore`; runtime dims `[Name, Operation, Resource]`). *Mitigation:* literals pinned from a live `list-metrics` read-back; a wrong namespace renders blank tiles / INSUFFICIENT_DATA (visible, not silent).
6. **Deploy-role least-privilege gaps.** Each new service surfaced missing deploy-role perms (the recurring add-action → re-apply-bootstrap → re-deploy loop). *Mitigation:* grants landed iteratively (vended-logs delivery API, data-protection, App-Signals read-back, EnableInsightRules, GetMetricData, a 6144-policy split); expect 1–2 more if new services are added.
