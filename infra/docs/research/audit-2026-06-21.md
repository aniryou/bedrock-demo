# Architecture Best-Practice Audit
### Bedrock AgentCore · Entra ID · Snowflake — Order-Triage Demo

**Date:** 2026-06-21
**Auditor scope:** Enterprise & AI architecture review across AWS (Bedrock AgentCore, ECR, IAM, CodeBuild/CI), Azure Entra ID (OAuth2 / On-Behalf-Of), and Snowflake (RBAC, RLS, auth, governance).
**Repos reviewed:** `bedrock-demo-infra`, `bedrock-demo-orchestrator`, `bedrock-demo-stubs`, `order-triage-agent`, `order-triage-knowledge`.

**Method:** Eight parallel domain auditors read the source directly; **every finding was independently re-read against the cited file by an adversarial verifier** to eliminate hallucinations and false positives, then a completeness critic swept for cross-cutting gaps. Of 84 raw findings, **11 were rejected** as unverifiable/false-positive and several severities were corrected downward where a compensating control existed.

> **Demo context.** This is a reference implementation, and the team has been disciplined about labelling simplifications in-code. Every severity below reflects that: *intentional demo shortcuts* are separated from *true defects that would ship if the stack were promoted as-is*.

## The five themes that tie the findings together

| # | Theme | Why it matters |
|---|-------|----------------|
| **A** | **The documented authorization model is not in the request path** | The Gateway + Cedar policy engine + OBO impersonation are provisioned and described as "the sole authorization layer," but the agent calls backends *directly* via public Lambda URLs with a static key. The governance story is currently decorative. |
| **B** | **Secrets live in state, stdout, env and docs — long-lived, unrotated** | The Entra OBO client secret is persisted in Terraform state; private keys and secrets are printed to stdout; API keys are plaintext runtime env; real tenant/account IDs and a personal email are committed in docs. |
| **C** | **IAM over-grant top to bottom** | The CI deploy role defaults to `AdministratorAccess` and is assumable from *any* branch/PR; runtime/gateway/KB roles use service-`*` wildcards on `Resource:"*"`; service trust policies lack confused-deputy conditions. |
| **D** | **Supply-chain reproducibility gaps** | The image is built with `pip` ignoring the committed `uv.lock`; only mutable `:latest` is published; base image and GitHub Actions are pinned by floating tag, not digest/SHA; the container runs as root. |
| **E** | **Missing baseline controls** | No Bedrock Guardrails anywhere; no CMK on Memory/KB; no CloudWatch log retention; no Snowflake network policy / statement timeout; no masking on credit data; unhardened artifact/state S3 buckets. |

## 🔴 High-severity findings

#### 1. GitHub OIDC deploy role grants AdministratorAccess

- **Severity:** 🔴 High  ·  **Domain:** AWS IAM & least privilege  ·  **Confidence:** high
- **Location:** [../../bootstrap/github_oidc.tf:30-34, 142-145](../../bootstrap/github_oidc.tf:30)
- **Issue:** The CD deploy role assumed by bedrock-demo-infra's GitHub Actions defaults to the AWS-managed AdministratorAccess policy. Any workflow run (or anyone who can push a workflow/branch in the repo) that assumes this role obtains full account control. This is explicitly labelled a demo default, so it is demo-acceptable, but it is a true defect for production: combined with the broad OIDC sub trust (see separate finding) it is the single highest-blast-radius grant in the stack.
- **Best practice:** IAM least privilege: CI/CD deploy roles should be scoped to the specific services Terraform manages (Bedrock AgentCore, ECR, S3, IAM, Lambda, Secrets Manager, CloudWatch/X-Ray), not AdministratorAccess. Avoid attaching AWS-managed AdministratorAccess to any automation principal.
- **Fix:** Author a least-privilege managed/customer policy enumerating only the services this Terraform touches and set deploy_policy_arn to it. If admin-equivalent breadth is unavoidable for terraform, at minimum add a permissions boundary and an IAM condition restricting which roles can be passed/created. Track removal of the AdministratorAccess default as a release gate.
- **Verifier note:** No correction needed. Evidence matches exactly. Note for completeness: the AdministratorAccess is the configurable default (var.deploy_policy_arn), so it can be remediated by passing a least-privilege policy ARN without code changes, and the repo's own comments/CD-SETUP.md already flag this as a demo default to scope down before production. Severity remains High given the blast radius if shipped as-is.

#### 2. GitHub OIDC trust pins repo but not branch/environment (sub uses :*)

- **Severity:** 🔴 High  ·  **Domain:** AWS IAM & least privilege  ·  **Confidence:** high
- **Location:** [../../bootstrap/github_oidc.tf:129-134 (deploy_trust), 68-72 (publish_trust)](../../bootstrap/github_oidc.tf:129)
- **Issue:** Both OIDC trust policies match the sub claim with repo:<org>/<repo>:* — i.e. ANY branch, tag, pull_request, or environment in the repo can assume the role. For the deploy role this means any commit on any branch (e.g. a feature branch, or a PR from a fork if pull_request triggers are enabled) can assume an AdministratorAccess role. The aud is correctly pinned to sts.amazonaws.com, and the repo is pinned, but the missing branch/environment constraint is the classic GitHub OIDC privilege-escalation gap. Demo-acceptable but a true production defect, especially paired with the AdministratorAccess default.
- **Best practice:** GitHub OIDC trust: pin the sub claim to a specific ref or environment (repo:org/repo:ref:refs/heads/main or repo:org/repo:environment:prod), not repo:org/repo:* . Use a deployment environment with required reviewers for the high-privilege deploy role.
- **Fix:** For the deploy role, change the sub condition to a protected-environment form, e.g. repo:org/bedrock-demo-infra:environment:production (and configure that environment with required approvers), or pin to ref:refs/heads/main. Keep publish scoped to main/tags as appropriate. This is independent of the policy-breadth fix and should be done even before AdministratorAccess is narrowed.

#### 3. Agent client secret stored in plaintext Terraform state and exposed as a TF output

- **Severity:** 🔴 High  ·  **Domain:** Azure Entra ID, OAuth & On-Behalf-Of flow  ·  **Confidence:** high
- **Location:** [../../entra/outputs.tf:26-29](../../entra/outputs.tf:26)
- **Issue:** The generated agent client secret is emitted as a Terraform output (marked sensitive, so masked in CLI logs) but is still written in cleartext into the entra/ state file, which versions.tf documents as LOCAL state (not the S3 backend). The consuming AWS stack also takes the secret as TF_VAR_entra_agent_client_secret and its variables.tf:135 comment explicitly says '(Stored in state; rotate before 2026-12-17.)'. So the impersonation secret lives in plaintext in at least one local state file on disk and in the awscc provider's state. Anyone with read access to the state file obtains a credential that can mint OBO tokens for any signed-in user. Demo-acceptable on a single operator laptop, but a real exposure in any shared/CI context.
- **Best practice:** Secrets must not persist in Terraform state; sensitive outputs only mask CLI display, not state-at-rest. Use a secret store (AWS Secrets Manager / Azure Key Vault) and reference it, encrypt state at rest, and restrict state access.
- **Fix:** Do not output the secret; instead create it out-of-band in a secret manager and reference it. If Terraform must create it, store the value directly into AWS Secrets Manager / Azure Key Vault, encrypt the state backend, restrict state ACLs, and rotate the secret after any state-file exposure.
- **Verifier note:** Minor correction: the AWS (terraform/) S3 backend sets encrypt = true, so the cleartext-at-rest concern applies most directly to the entra/ LOCAL state file (no backend, no encryption) and to anyone with read access to either state. Severity remains High for any production/shared/CI deployment (which the S3 remote backend implies is the intended target), but on a single isolated operator laptop the practical exposure is closer to Medium, consistent with the finding's own demo caveat.

#### 4. AgentCore Gateway + Cedar policy engine are provisioned but bypassed: all tool calls go direct to public Lambda URLs

- **Severity:** 🔴 High  ·  **Domain:** Cross-cutting / completeness  ·  **Confidence:** high
- **Location:** [../../terraform/runtime.tf:25-28](../../terraform/runtime.tf:25)
- **Issue:** Both runtimes inject GATEWAY_URL but also inject SAP_API_URL/ORDER_ACTIONS_URL/SNOWFLAKE_DATA_URL set to the raw public Lambda Function URLs (authorization_type=NONE). The agent tools (sap.py, orders.py, data.py) call cfg.sap_api_url / cfg.order_actions_url / cfg.snowflake_data_url directly via httpx with a static X-API-Key; config.use_gateway() is defined but never called anywhere in the agent source or tests. So the AgentCore Gateway and its Cedar policy engine — described in policy.tf as 'the sole authorization layer' and in order_actions_stub/app.py as enforcing 'who may flag' — are never in the request path. Every read AND the state-changing flag are authorized by nothing but a shared static key. Production impact: the entire documented authorization model (Cedar agent-permission ∩ backend user-permission) is non-functional; this is also why the public-Function-URL findings are higher risk than rated, since there is no compensating control. Demo-acceptable only if the team knows the Gateway is currently decorative; as written the README/ADR claim authorization the deployment does not perform.
- **Best practice:** AgentCore Gateway should be the only ingress to backend tools; backend Lambda Function URLs should not be directly reachable by the runtime (and ideally not public). If Cedar is the authorization layer, the data path must traverse the Gateway.
- **Fix:** Either (a) route tool calls through the Gateway (use the gateway_url + an MCP/HTTP client, make use_gateway() live, set the backend env URLs to the Gateway target URLs), and make the Lambda Function URLs non-public (IAM auth, invokable only by the Gateway role); or (b) if direct-to-Lambda is intentional for the demo, remove the Cedar/Gateway resources and the docs claiming they authorize calls, and document that authorization is the static API key only.
- **Verifier note:** Finding is accurate as written. One precision: the Gateway's OpenAPI targets (gateway.tf, snowflake_lambda.tf) also rewrite their backend to the same raw Lambda Function URLs, so the intended design is agent -> Gateway (Cedar + X-API-Key injection) -> Lambda. The defect is purely that runtime.tf points the agent's *_API_URL env vars at the Lambda URLs instead of the Gateway target URL, and the agent never uses gateway_url/use_gateway. README.md lines 67-68/110-111/118-119 explicitly (and incorrectly) state SAP_API_URL/ORDER_ACTIONS_URL "point at Gateway," directly contradicting the deployed wiring, strengthening the documentation-vs-deployment mismatch claim.


## 🟠 Medium-severity findings

#### 1. Stub Lambda Function URLs are public (authorization_type=NONE, InvokeFunctionUrl to principal="*")

- **Severity:** 🟠 Medium  ·  **Domain:** AWS IAM & least privilege  ·  **Confidence:** high
- **Location:** [../../terraform/snowflake_lambda.tf:60-71 (snowflake); sap_lambda.tf:21-32; order_actions_lambda.tf:23-34](../../terraform/snowflake_lambda.tf:60)
- **Issue:** All three backend Lambdas (SAP, order-actions, Snowflake-query) expose Function URLs with authorization_type=NONE and an aws_lambda_permission granting lambda:InvokeFunctionUrl to principal="*" with function_url_auth_type="NONE". These are unauthenticated public internet endpoints; the only access control is a static X-API-Key the application checks. The Snowflake endpoint in particular fronts a Lambda that holds key-pair credentials to Snowflake, so anyone on the internet who learns/guesses the key reaches the data path directly, bypassing the Gateway, the Cedar policy engine, and (on the OBO path) user impersonation. Demo-acceptable as a stub, but in production this defeats the entire RBAC/RLS/OBO governance story and is the most likely real-world exposure vector.
- **Best practice:** IAM least privilege / network exposure: do not expose data-backing Lambdas as unauthenticated public Function URLs. Use authorization_type=AWS_IAM so only the Gateway/runtime role (a scoped principal) can invoke, or front with API Gateway + a private VPC link; never grant InvokeFunctionUrl to principal="*".
- **Fix:** Set authorization_type=AWS_IAM on the Function URLs and replace the principal="*" permission with a permission scoped to the gateway role ARN (the Gateway can SigV4-sign outbound). If a static-key path is required for the demo, at minimum scope the lambda:InvokeFunctionUrl principal to the gateway/runtime role rather than "*", and document that the key is the sole control.
- **Verifier note:** Severity reduced from High to Medium. The finding correctly self-identifies this as a documented demo with stub services (repo README labels them "dummy back-office services"), and the SAP/order-actions stubs hold only deterministic in-memory data — for those two the public-URL risk is low. The genuine exposure is the snowflake Lambda, which reads live Snowflake data, but it does so via a SELECT-only AGENT_RO role through key-pair JWT, and the static X-API-Key is a non-trivial credential. The risk is a legitimate production-hardening concern (it does bypass Gateway/Cedar/OBO if URL+key leak) but is an intentional, clearly-documented demo simplification with a read-only scoped backing role, which places it at Medium rather than High. The finding does not overclaim — it explicitly says "Demo-acceptable as a stub, but in production..." — so it stays is_real=true at Medium.

#### 2. Same plaintext secrets repeated on the OBO runtime

- **Severity:** 🟠 Medium  ·  **Domain:** AWS Bedrock AgentCore (runtime, gateway, memory, identity, KB)  ·  **Confidence:** high
- **Location:** [../../terraform/runtime_obo.tf:71-73](../../terraform/runtime_obo.tf:71)
- **Issue:** The parallel CUSTOM_JWT (OBO) runtime repeats the same plaintext SNOWFLAKE_API_KEY / SAP_API_KEY / PRINCIPAL env injection as the primary runtime. This is the same defect as runtime.tf and doubles the exposure surface (a second runtime config carrying the keys). It is especially notable on the OBO runtime, whose whole purpose is to scope access to the impersonated user via short-lived OAuth tokens — yet it still ships long-lived static backend keys in cleartext.
- **Best practice:** Bedrock AgentCore: do not pass long-lived static secrets as runtime env; route backend auth through Identity credential providers / OBO token exchange only.
- **Fix:** Drop the static API keys from the OBO runtime env and rely on the OBO credential provider + Gateway-injected credentials. Keep only non-secret config (model id, IDs, URLs, OBO flags) as env.
- **Verifier note:** Severity reduced High -> Medium. Secrets are passed via sensitive Terraform variables (TF_VAR/.env), not hardcoded plaintext in the repo, so there is no committed-secret leak. Real residual risk is that the keys land as AgentCore runtime env on a second runtime (env is not a secrets store), doubling the surface. Recommendation stands for prod hardening (route backend auth through Identity/OBO credential providers, keep only non-secret config as env), but note this is an intentional, count-guarded demo simplification rather than a critical exposure.

#### 3. No Bedrock Guardrail attached to any model invocation path

- **Severity:** 🟠 Medium  ·  **Domain:** AWS Bedrock AgentCore (runtime, gateway, memory, identity, KB)  ·  **Confidence:** high
- **Location:** [../../terraform/runtime.tf:19-32](../../terraform/runtime.tf:19)
- **Issue:** Neither runtime, nor the KB (RetrieveAndGenerate), nor the Memory strategy model invocations attach a Bedrock Guardrail (no aws_bedrock_guardrail resource exists anywhere in the repo; the only 'guardrails' reference is a prose business rule in ../../../agent/src/order_triage/agent.py:25). This agent reads customer/order data from Snowflake and can flag orders, so the model sees and emits PII and business-sensitive content with no content filtering, PII redaction/blocking, denied-topic, or prompt-injection (contextual grounding) protection. The runtime IAM grants bedrock:InvokeModel directly, so model calls run ungated. Demo-acceptable for a sandbox, but for a PII-handling production agent the absence of any Guardrail is a real control gap.
- **Best practice:** Bedrock best practice: attach a Bedrock Guardrail (content filters, PII entity handling, denied topics, contextual grounding) to InvokeModel / RetrieveAndGenerate, especially for agents processing PII or taking actions.
- **Fix:** Define an aws_bedrock_guardrail (and version) with PII filters and prompt-attack contextual grounding, surface its identifier+version to the agent (e.g. GUARDRAIL_ID/GUARDRAIL_VERSION env), and pass guardrailConfig on every Converse/InvokeModel and on the KB RetrieveAndGenerate call.
- **Verifier note:** Finding is real but severity should be Medium, not High. This is an absence-of-a-control / defense-in-depth gap (no content filters, PII handling, denied topics, or contextual-grounding prompt-injection protection), not an actively exploitable vulnerability in the current artifact. The repo is explicitly a demo (comments note "demo", "once the deploy is unblocked", and MODEL_PROVIDER defaults to the local anthropic path), and the finding itself concedes it is "demo-acceptable for a sandbox." Attaching a Bedrock Guardrail and passing guardrailConfig on Converse/InvokeModel and KB RetrieveAndGenerate is a valid production hardening recommendation, but the conditional/prospective nature of the impact makes Medium the appropriate severity rather than High.

#### 4. Runtime IAM role grants wildcard actions and resources far beyond least privilege

- **Severity:** 🟠 Medium  ·  **Domain:** AWS Bedrock AgentCore (runtime, gateway, memory, identity, KB)  ·  **Confidence:** high
- **Location:** [../../terraform/iam.tf:26-35](../../terraform/iam.tf:26)
- **Issue:** The runtime execution role allows bedrock-agentcore:* (every AgentCore action: create/delete gateways, memories, policies, registries, runtimes) and logs:* on Resource="*", plus bedrock:InvokeModel on all models. The header comment (iam.tf:1) acknowledges 'Broad demo policies — tighten to least privilege for production', so this is flagged as a known demo simplification, but it is a genuine privilege-escalation risk: a compromised agent container could reconfigure or delete the entire AgentCore control plane and the Cedar policy engine that gates its own tool calls.
- **Best practice:** IAM least privilege: scope AgentCore actions to the specific runtime/memory/gateway data-plane actions (e.g. GetWorkloadAccessTokenForJWT, GetResourceOauth2Token, InvokeGateway), bedrock:InvokeModel to the specific model ARNs, and logs to the runtime's own log-group ARNs.
- **Fix:** Replace bedrock-agentcore:* with the explicit data-plane actions the agent uses; scope logs:* to CreateLogStream/PutLogEvents on the agent's log-group ARN; scope bedrock:InvokeModel to the configured model + embedding model ARNs.

#### 5. Gateway IAM role grants bedrock-agentcore:* and secretsmanager:GetSecretValue on Resource=*

- **Severity:** 🟠 Medium  ·  **Domain:** AWS Bedrock AgentCore (runtime, gateway, memory, identity, KB)  ·  **Confidence:** high
- **Location:** [../../terraform/iam.tf:60-65](../../terraform/iam.tf:60)
- **Issue:** The Gateway role allows lambda:InvokeFunction(Url), bedrock-agentcore:*, and secretsmanager:GetSecretValue on Resource="*". The Gateway only needs to invoke the three specific stub Lambdas and read the two api-key credential-provider secrets. Granting GetSecretValue on every secret in the account, plus the full AgentCore action surface, is well beyond what a Gateway needs and would let a compromised Gateway role read unrelated secrets (e.g. the Snowflake key-pair secret order-triage/snowflake). Demo-acceptable per the module's stated intent, but a real over-grant.
- **Best practice:** IAM least privilege: scope lambda invoke to the three target function ARNs, secretsmanager:GetSecretValue to the bedrock-agentcore-identity!* credential-provider secret ARNs only, and replace bedrock-agentcore:* with the specific Gateway actions.
- **Fix:** Restrict Resource to the specific Lambda ARNs and the credential-provider secret ARN prefix (bedrock-agentcore-identity!*), and narrow the AgentCore action list.

#### 6. Entra agent client secret persisted in Terraform state

- **Severity:** 🟠 Medium  ·  **Domain:** AWS Bedrock AgentCore (runtime, gateway, memory, identity, KB)  ·  **Confidence:** high
- **Location:** [../../terraform/variables.tf:131-136](../../terraform/variables.tf:131)
- **Issue:** entra_agent_client_secret is marked sensitive and flows into awscc_bedrockagentcore_o_auth_2_credential_provider.entra_obo (identity.tf:43-49). The variable's own description admits 'Stored in state; rotate before 2026-12-17.' Even though it is marked sensitive (kept out of plan output) and the S3 backend has encrypt=true (versions.tf:23), the cleartext secret lives in remote state, so anyone with read access to the state bucket can recover it. The credential-provider itself stores the secret securely in AgentCore-managed Secrets Manager, so the state copy is an avoidable secondary exposure. Demo-acceptable but a real defect for production.
- **Best practice:** Avoid passing long-lived OAuth client secrets through Terraform variables/state; source them at apply time from a secret store, or use a write-only/ephemeral mechanism, and lock down state-bucket access.
- **Fix:** Feed the client secret to the credential provider via a Secrets Manager reference (data source / ephemeral resource) rather than a plain variable so it is not materialized in state, restrict state-bucket access, and document a rotation runbook.

#### 7. Agent CI exposes ANTHROPIC_API_KEY and SKILLS_TOKEN to untrusted fork pull requests

- **Severity:** 🟠 Medium  ·  **Domain:** ECR, container build, CodeBuild, CI/CD & supply chain  ·  **Confidence:** high
- **Location:** [../../../.github/workflows/agent-ci.yml:7-20](../../../.github/workflows/agent-ci.yml:7)
- **Issue:** The workflow triggers on a bare `pull_request:` (no branch filter, no environment gating) and sets ANTHROPIC_API_KEY and SKILLS_TOKEN as job-level env vars from secrets. For public/forkable repos, `pull_request` from a fork still injects repository secrets into the job environment, and the PR author controls the code that runs (tests, the `make skills` step, ruff config, conftest). A malicious PR can trivially echo or POST $ANTHROPIC_API_KEY / $SKILLS_TOKEN (the latter has read access to the private knowledge repo). This is a genuine secret-exfiltration path, not merely demo-acceptable, if the repo is ever public or accepts external contributors.
- **Best practice:** Never expose long-lived secrets to workflows triggered by untrusted PRs. Gate secret-bearing jobs behind a GitHub Environment with required reviewers, or split into a no-secret PR-validation job and a separate push/label-gated job that holds the secrets. The eval gate that needs ANTHROPIC_API_KEY should run on push to main / a protected environment, not on fork PRs.
- **Fix:** Split ci.yml: keep lint + hermetic unit tests on `pull_request` with NO secrets; move the eval gate (which needs ANTHROPIC_API_KEY) and skills fetch (SKILLS_TOKEN) into a job that runs only on `push: main` or behind a protected `environment:`. Alternatively restrict the PR trigger so secret-bearing steps run only for same-repo branches (`if: github.event.pull_request.head.repo.full_name == github.repository`).
- **Verifier note:** Finding is real and evidence is accurate. Severity adjusted High -> Medium: secret exposure to fork PRs is not GitHub's default (requires maintainer-enabled fork-secret sharing + the repo being public/forkable), so exploitability is conditional. Same-repo branch PRs always receive secrets but the author is already trusted. Remediation (split secret-bearing eval/skills steps behind push:main or a protected environment, or add an `if: github.event.pull_request.head.repo.full_name == github.repository` guard) is correct and recommended before this repo is made public or opened to external contributors.

#### 8. Deployed image is built with `pip install` and ignores the committed uv.lock — not reproducible

- **Severity:** 🟠 Medium  ·  **Domain:** ECR, container build, CodeBuild, CI/CD & supply chain  ·  **Confidence:** high
- **Location:** [../../../agent/Dockerfile:16](../../../agent/Dockerfile:16)
- **Issue:** A `uv.lock` is committed at the repo root (and CI/Makefile use `uv sync`), but the production image installs with `pip install --no-cache-dir ".[deploy]"` against unpinned ranges in pyproject.toml (e.g. `strands-agents>=0.1.0`, `boto3>=1.35.0`, `bedrock-agentcore>=0.1.0`, `aws-opentelemetry-distro>=0.10.0`). The image resolves transitive deps fresh at build time, so the same source can produce different dependency closures over time, and the lockfile that CI validated is never applied to the shipped artifact. This defeats supply-chain reproducibility and means a bad upstream release can silently land in the runtime image.
- **Best practice:** Build the deployed image from the lockfile (`uv sync --frozen --no-dev` / `uv pip install --require-hashes`) so the image dependency set is identical to what CI tested and is pinned/hash-verified.
- **Fix:** Install uv in the image and run `uv sync --frozen` (or `uv pip install --no-deps -r` a compiled requirements file with hashes) using the committed lock, including the deploy extras. Keep pyproject ranges for development but ship the locked closure.

#### 10. Agent app uses a long-lived client secret instead of a certificate / federated credential

- **Severity:** 🟠 Medium  ·  **Domain:** Azure Entra ID, OAuth & On-Behalf-Of flow  ·  **Confidence:** high
- **Location:** [../../entra/main.tf:96-100](../../entra/main.tf:96)
- **Issue:** The middle-tier (agent) app authenticates to the Entra token endpoint with a client secret whose default lifetime is 2 years (17520h, entra/variables.tf:37-41; the az path uses SECRET_YEARS=2, entra_provision.py:57). In production Microsoft and enterprise baselines recommend a certificate or a workload-identity federated credential (no stored secret), or at minimum a short (90-180 day) rotated secret. Demo-acceptable for a reference build, but a true defect for production.
- **Best practice:** Prefer certificate credentials or federated identity credentials (managed identity / workload identity federation) for confidential OAuth clients; if a secret must be used, keep its lifetime short and rotate it. Avoid multi-year client secrets.

- **Verifier note:** Severity Medium is at the upper edge of defensible. This is a parameterized default (not a hardcoded secret) in a clearly-labeled reference/demo build, and the secret value is marked sensitive and not committed to source. A grader could reasonably score this Low. The remediation is sound: switch to azuread_application_certificate or azuread_application_federated_identity_credential to eliminate the symmetric secret, or cut secret_end_date_relative / ENTRA_SECRET_YEARS to ~90 days with a rotation runbook.

#### 11. Headless bootstrap admin is granted ACCOUNTADMIN as DEFAULT_ROLE and all DDL runs as ACCOUNTADMIN

- **Severity:** 🟠 Medium  ·  **Domain:** Snowflake RBAC, RLS, auth & data governance  ·  **Confidence:** high
- **Location:** [../../scripts/gen_bootstrap_admin.sh:50-51](../../scripts/gen_bootstrap_admin.sh:50)
- **Issue:** The generated bootstrap admin is granted ACCOUNTADMIN and ACCOUNTADMIN is set as its DEFAULT_ROLE, and every setup/rls/test_user script begins with USE ROLE ACCOUNTADMIN while connect_admin() hardcodes role=ACCOUNTADMIN. #### 12. X-API-Key check is bypassed by any non-empty Authorization: Bearer header

- **Severity:** 🟠 Medium  ·  **Domain:** Snowflake RBAC, RLS, auth & data governance  ·  **Confidence:** high
- **Location:** [../../../stubs/snowflake_stub/app.py:46-61](../../../stubs/snowflake_stub/app.py:46)
- **Issue:** _authorize() returns the bearer token and skips _check_key() entirely whenever a non-empty 'Authorization: Bearer <x>' header is present. The app never validates that bearer itself — it relies on Snowflake to reject a bad token downstream. Combined with the public Function URL (auth NONE), this means an unauthenticated internet caller who omits X-API-Key but sends ANY bearer string reaches the Snowflake OAUTH code path. Snowflake will reject a forged/expired token, so data isn't directly exposed, but the API-key gate (the app's stated auth boundary) is trivially circumvented, every such request consumes a warehouse round-trip (DoS/cost vector), and a stolen/replayed real user token is honored with no app-side checks (no audience/expiry/issuer validation at the edge). The bearer-wins-over-API-key precedence is intentional for OBO, but it should still require a valid API key as the transport credential in addition to the bearer.
- **Best practice:** Defense in depth: a transport/API credential and a user-identity token are orthogonal; presenting a user token must not waive the API-key (caller-authenticity) check. Validate forwarded bearer tokens (issuer/audience/expiry) at the edge rather than relying solely on the downstream system.
- **Fix:** Always enforce _check_key(x_api_key) first, then independently parse the bearer for the OBO path: e.g. call _check_key() unconditionally, and only after it passes decide service-vs-user based on the presence of a bearer. Optionally validate the JWT's iss/aud/exp before forwarding to Snowflake.
- **Verifier note:** Finding is accurate and not misquoted. Severity reduced High -> Medium: the bearer branch bypasses the app's X-API-Key gate and enables unauthenticated cost/DoS (every request hits Snowflake via the whoami probe in _set_served), but it does NOT directly expose data because Snowflake validates the OAUTH token downstream. The recommendation (enforce _check_key() unconditionally, then branch service-vs-user on bearer presence, optionally validating iss/aud/exp before forwarding) is sound. Note this is a demo stub repo, though the production Terraform that deploys the public auth-NONE Function URL is present in ../../terraform/snowflake_lambda.tf, so it is not purely a local-dev convenience.

#### 13. No prompt-injection defense: untrusted KB/Snowflake/ontology text flows into the model, and model output drives a state-changing tool

- **Severity:** 🟠 Medium  ·  **Domain:** Agent application code & runtime security  ·  **Confidence:** high

## Cross-cutting deep-dive: will routing tool calls through the Gateway work with the OBO flow? (re: H4)

**Today they are mutually exclusive identity paths.**

| | Inbound auth | Backend sees | Carries user identity? |
|---|---|---|---|
| **Gateway path** | `authorizer_type = "AWS_IAM"` (runtime SigV4) | Static `X-API-Key` injected by an **api_key** credential provider on all three targets | ❌ service identity only |
| **OBO path** | `custom_jwt_authorizer` on the **runtime** (Entra user JWT) | Per-user `Authorization: Bearer` minted in agent code, sent **direct to the Lambda URL** | ✅ the human |

The OBO exchange (`GetWorkloadAccessTokenForJWT` → `GetResourceOauth2Token` with `ON_BEHALF_OF_TOKEN_EXCHANGE`) is performed **imperatively by the agent at runtime**; `identity.tf` states OBO is *"a RUNTIME flow … not a provider setting."* The Gateway is never in the OBO path.

**Consequence:** naively repointing the tools at the *current* Gateway would route Snowflake calls through the api_key provider and arrive as the service `AGENT_RO` role — **collapsing per-user RLS**, the opposite of OBO.

**Verified capability (AWS docs, 2026-06-21):** AgentCore Gateway egress **does** support declarative silent OBO. A target's OAUTH credential provider accepts `grantType` ∈ `CLIENT_CREDENTIALS | AUTHORIZATION_CODE | TOKEN_EXCHANGE`; **`TOKEN_EXCHANGE`** is documented verbatim as *"the gateway exchanges the inbound user's access token for a target-scoped token … without triggering additional consent flows."* The gateway service role (not agent code) is granted `GetWorkloadAccessToken`/`GetResourceOauth2Token`. **Critical constraint:** OBO egress is supported on **MCP-server and OpenAPI-schema targets only — NOT Lambda / API-Gateway targets.** The three targets here are OpenAPI targets (`open_api_schema.inline_payload`) backed by Lambda Function URLs, so they qualify.

**Therefore the H4 + OBO fix (ADR-0001 target state), split by `authority`:**
- **Service-authority reads → through the Gateway** with the existing api_key (or `CLIENT_CREDENTIALS`) provider. Cedar back in the path; Lambda URLs no longer hit directly.
- **User-authority calls → Gateway-mediated OBO:** flip the Gateway inbound to `CUSTOM_JWT`, and change the Snowflake target's credential provider from `api_key` to **`OAUTH` + `grantType=TOKEN_EXCHANGE`** referencing the `entra_obo` provider. The Gateway then mints the per-user token itself — **`identity_obo.py` becomes unnecessary for the Snowflake path.**
- **Alternative:** `JWT_PASSTHROUGH` (requires `AUTHENTICATE_ONLY` inbound) forwards the caller's bearer unmodified — useful if you prefer the agent to keep minting.
- Cedar must move from `IamEntity` → `OAuthUser` either way.

**Net:** routing through the Gateway is **fully compatible with OBO** and is the cleaner design — it removes the bespoke runtime token-exchange code. The only real work is the inbound-auth flip (`AWS_IAM` → `CUSTOM_JWT`), the per-target credential-provider swap, and the Cedar rewrite.
