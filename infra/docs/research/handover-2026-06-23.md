## What it took to land the cutover (lessons — likely to recur on re-deploy)
- **The "SCP blocks Bedrock" was a myth.** The gated `infra.preflight` probed `bedrock:ListFoundationModels`, which the least-privilege deploy role (H1/H2) doesn't grant → it failed on the role's own IAM and *mis-reported an SCP block*. Fixed by re-pointing the probe to `bedrock:ListKnowledgeBases` (granted + used by the apply) — infra **#36** (`2ed9279`). Preflight then PASSED → Bedrock was never SCP-denied.
- **Iterative least-privilege deploy-role discovery.** The first real apply surfaced perms the generated `deploy_perms` lacked (the policy comment always anticipated "refine from a real CloudTrail"). Added across **#40** (`1223e5f`) + **#42** (`576e56d`): `s3:GetObjectTagging`/`s3:DeleteObject`, `secretsmanager:GetResourcePolicy`, `logs:{Describe,Put,Delete}ResourcePolicies`, `cloudformation:{Get,GetResourceRequestStatus,Create,Update,Delete,List}Resource` (awscc Cloud Control), `lambda:GetFunctionCodeSigningConfig`. **If you add resources that touch new services, expect 1–2 more of these — same loop: add action → re-apply bootstrap → re-deploy.** Recurred on the memory-observability deploy (run 27961168020): added `logs:ListTagsForResource`/`DeleteLogGroup`/`UntagResource` + the vended-logs delivery API (`Put/Get/Delete DeliverySource`+`DeliveryDestination`, `Create/Get/Delete/Update Delivery`) in infra **#68** — same loop.
- **Cedar `when { true }` is rejected.** The policy engine's semantic validation (FAIL_ON_ANY_FINDINGS) flags a provably-true condition as "Overly Permissive". Fixed to `principal.hasTag("scp")` — infra **#44** (`599e097`). **JWT claims are exposed as Cedar TAGS** (`principal.hasTag/getTag`), NOT `principal.claims[...]`.
- **ROPC `AADSTS50126`.** The `svc-triage-test` account needed a *permanent* password (not temp/must-change) and no MFA/Security-Defaults block (ROPC = legacy password grant). Resolved by the user; `ENTRA_TEST_USER`/`ENTRA_TEST_PASSWORD` now in `.env`.

## Gotchas (cost time — save yourself)
- **`.env*` is permission-guarded** for the agent (can't Read/Edit, can't `source` it in a Bash tool call — the classifier blocks routing around the guard). The **user** must set/source `.env` values themselves (e.g. the ROPC creds). Plain `source ../.env` doesn't export — use `set -a; source ../.env; set +a`.
- **`bedrock-demo-orchestrator/scripts/lib.sh` is bash** (`BASH_SOURCE`, `set -euo pipefail`); run via `bash -c 'source scripts/lib.sh && <fn>'`, not zsh. Sourcing `.env` shell-expands values → passwords with `$`/backtick/`!` get mangled (single-quote them in `.env`; `seed_entra_secret` greps the client secret to avoid this).
- **terraform needs ≥1.10** — local default is 1.5.7; use `~/bin/terraform` (1.10.5, from `make prereqs`). `validate` needs AWS creds + provider init.
- **Deploy-role least-privilege is intentionally tight** — a brand-new resource type may need a new IAM action (see "lessons"); discover via the failed apply's `AccessDenied`, add to `bootstrap/github_oidc.tf` `deploy_perms`, re-apply bootstrap.
- **Workflow (multi-agent) sub-agents can edit files** — use `agentType: "Explore"` or read-only prompts + `git status` after, for analysis runs.
- **Bulk `rm -rf` is permission-denied** — prefer `git rm`.
- All gateway-only/C5b work **and the post-deploy cleanup** are **merged to `main`** in every repo (no lingering feat branches); `main` == deployed — **except** #50's deploy-role IAM tightening, where the code is merged but the *live* role lags until bootstrap is re-applied (see Open items).
